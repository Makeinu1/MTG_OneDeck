# 理想のMTGエンジンへ向けた設計手法(engine-design-method)

> メタ文書。**何を作るか**(=`docs/architecture-substrate-compiler.md`)の一段上、**どう作るか=
> どう設計を磨き続けるか**を定める。実装の前に置く規律であり、各反復で本書自身も更新する。
> 最終更新: 2026-06-23。

## 0. 中心命題 — なぜ「実装」より先に「サイクル」が来るか

ゴールは **「MTGオラクル文法をコンパイルするとエンジン駆動に乗る」**。そこへ至る道は、機能を
実装することではなく、**「ルールを理解して、それをエンジンの状態(変数)へどう落とすか」という
サイクルを磨き続けること**。実装フェーズはサイクルが収束した後に来る。

根拠(計測が示す):効果行の **87%は上位20の効果アトムで構成**(=語彙は足りている)のに、
今すぐ自動実行できるのは **14%**、構造化できているのは **58%**。ボトルネックは動詞でも語彙でもなく、
**「その動詞が読み書きする“エンジンの状態”が設計されていない」**こと。最も戻しにくいのは state 設計
(後から変数/層/ゾーンを足すと前方互換と層の再設計で最大の代償)。ゆえに **戻せない設計を、戻せる
紙の上で先に収束させる**。これが順序の核心。

## 1. サイクルが磨く対象 — 生きた2成果物

「磨き続ける」を雰囲気で終わらせないため、対象を2つの文書に固定する(どちらもコードではない)。

### ① エンジン状態オントロジー(ESO)
「MTGを再現するためにエンジンが持つべき変数の最小十分集合」。各エントリは次の列を持つ:

| 列 | 意味 |
|---|---|
| 変数/概念 | オブジェクト特性・ゾーン・ゲーム/プレイヤー変数・イベント語彙・層・juncture・SBA条件 等 |
| CR根拠(上から) | どの総合ルールが要求するか(CR条番号) |
| コーパス需要(下から) | 何枚のカードが実際に要求するか(頻度・代表カード) |
| state read/write | エンジンのどの状態を読む/書くか |
| 検証手段(物差し) | この正しさをどの oracle で確かめるか(§3) |
| trust | その物差しでどこまで言えるか(検証済/不一致/**検証不能**) |
| 既知欠落 | 意図的に落とす相互作用(誤謬予算の在処) |

### ② オラクル文法 ⇄ 状態変異カタログ
「ルールをエンジンにどう落とすか」の本体。各文法構文(効果アトム・誘発条件・期間・対象フィルタ・
モード・コスト節…)を、**ESO のどの変数の read/write 列に写すか**を1対1で定義する。これがコンパイラ
(IR→commands)の着地先契約になる。

## 2. 二面分析(設計の駆動力)

- **ルール面(top-down)**: CR から、ゲームが要求する state の最小集合を導く(オブジェクト/特性/ゾーン/
  イベント/層/タイミング/SBA)。
- **カードテキスト面(bottom-up)**: 17,491枚コーパスから、文法が**実際に要求する** state read/write・
  購読イベント・修整・期間・条件を、**頻度加重**で抽出する。既存 `npm run grammar-coverage` 系が器。
- **収束**: 両者を突き合わせる。下面は上面を**検証・優先順位づけ**する(CRのどの部分が高頻度=荷重か、
  どこが稀=後回しか)。上面は下面に**着地先の語彙**を与える。サイクルは、ESO が CR忠実 かつ
  カード分布の頭を覆う **最小十分モデル**へ収束するまで回す。

スライス順(一度に全部やらない・毎回コーパスで再検証):
1. 有効特性 + 層オントロジー(P/T・型・キーワード付与) 2. イベント語彙(誰の/何の事象/介在条件/頻度)
3. ゾーン + プレイヤー(別/共有/ダミー相手) 4. タイミング(フェイズ/ステップ/juncture)+ SBA。

## 3. 認識論 — 「完全な物差しは無い」を前提にする

正しさの計測には oracle(物差し)が要るが、**我々は完全な物差しを持っていない**。物差し自身が誤りうる。
従って:

- **検証(verification)ではなく反証(falsification)**。「正しいと証明」はできない。できるのは
  「反証の試行を増やしながら、まだ反証されていないを維持」(corroboration)。
- **KPIは単一スカラーの“正答率”にしない**(それは持っていない完全な物差しを暗に仮定する)。
- **複数の独立した不完全な物差しを突き合わせ、不一致を信号に、人間が裁定する。** 一致は弱い陽性、
  不一致は発見。物差し同士の**独立性(故障モードの違い)**が証拠価値を生む。
- **3状態を持つ**:`検証済`(物差しが当たり不一致なし)/`不一致`(物差しが割れた=裁定待ち)/
  **`検証不能`(どの物差しも当てられない=trust unknown)**。
  **鉄則:`検証不能` を緑(pass)に混ぜない。** 検証不能を黙って通すのが silent divergence の正体。
  常に「未検証」と可視化してユーザー/レビューへ見せる。

### 物差しの束(各々不完全・故障モードが異なる)

| 物差し | 信頼 | スループット | 主な故障モード |
|---|---|---|---|
| CR本文 + 人間裁定(gold) | 高 | 低 | 人為ミス・解釈差 |
| 参照エンジン差分(Forge/XMage) | 中 | 高 | 実装バグ |
| LLM挙動予測(無作為カード→Claude が判断した挙動 vs エンジン) | 中(頭)/低(尾) | 非常に高 | 角ケースで自信過剰に誤る |
| ゴールデン再生(コマンド列+期待盤面) | 著者依存 | 固定 | 著者の誤り |
| 内部不変条件 I1〜I12 | 高 | 高 | **腐敗しか見ない**(ルール誤りは見ない) |

### LLM-oracle の設計条件(主力だが扱いに条件)
- **相関エラーの遮断(最重要・凍結前は要件)**: Claude はエンジンの **state モデル設計者**でもある。
  盲点が共有されると「LLMとエンジンが一致」の情報量は薄い。LLM-oracle は**設計プロンプトと分離**する。
  **M-CONTRACT 凍結前には、少なくとも1つの非LLM独立物差し(Forge/XMage 差分 or 人間の盤面再生 gold)が
  代表カード集合に当たっていなければならない(助言ではなく要件・§5-4)**。LLM-oracle が*唯一*の独立物差し
  である状態は、それ自体を **tripwire**(相関エラーが物差しごと死ぬ)として扱う。
- **盲予測**: エンジン出力を見せず、オラクル文だけから挙動を予測(汚染防止)。
- **敵対シナリオ生成**: LLM は「このカードの厄介な相互作用は?」の生成器としても使う(=敵対サンプリング自動化)。

### CR を一次の決定論的権威に(deterministic と interpretive の弁別)— 2026-06-24 改訂
**最重要の運用規律**。M0 の問いは2種に分かれ、混同すると物差し(LLM)で答えを「予測」しようとして浪費・誤収束する。

- **決定論的(rules-defined)な問い** = 総合ルール(`rule/Magic_The_Gathering_Comprehensive_Rules.txt`)が**一意に**答える:
  ゾーン遷移(draw=121.1 library→hand / discard=701.9a hand→owner's graveyard / dies=700.4 battlefield→graveyard /
  destroy=701.8a battlefield→owner's graveyard / sacrifice=701.21a battlefield→owner's graveyard)、owner vs controller(108.4/110)、
  キーワードの定義、SBA(704)、ターン構造(500番台)等。**ここに LLM を当てるのは純粋なノイズ**(プロンプト次第で答えが揺れる)。
- **解釈的(recognition/曖昧)な問い** = カード文の**認識**(この prose は sacrifice を発動しているか/`their` は相手か/scope は each-player か you か)。
  ここは独立オラクル(相関遮断)が価値を持つ。

**権威の順序 = CR(決定論) > 人間 gold > LLM-oracle(解釈・相関遮断のみ)。** 帰結する鉄則:
1. **決定論的な軸・写像は CR から真理テーブルを引いて定義する**(`docs/` に CR 引用付きで固定)。分類器・gold・**オラクルの prompt** の
   三者すべてを CR に anchor する(オラクルには CR の写像を明示=ノイズ源を断つ)。
2. **オラクルと分類器が決定論的軸で割れたら、tiebreaker は CR**(プロンプト再走ではない)。**不一致を見たら、まず「決定論的か」を判定**し、
   そうなら CR を引いて終了する。**precedent(2026-06-24)**: Slice3 iter3-b で「sacrifice は graveyard に触れるか」をオラクルで判じようとして
   prompt を3回再走(v3/v3.1/v3.2)し誤った方向(除外)へ収束しかけた。CR 701.21a 一行(owner's graveyard へ移動)で即決し、3回は全て不要だった。
3. **flip-flop(§7)の射程を限定**: 外部真理(CR)が在る決定論的問いは、**分類器とオラクル prompt を同時に CR へ寄せてよい**(交絡しない=参照が CR)。
   flip-flop の「一変数ずつ」は**外部真理の無い解釈的問い**(物差し校正)に限る。
4. **CR 真理テーブル = ゲート条件4(非LLM独立物差し)の canonical 実体**。Forge/XMage 差分より上位の権威。CR 由来の決定論的期待値に対する
   **CR-conformance 監査**(分類器・gold・runtime が CR 条文と一致するか)を凍結前の要件に接続する。LLM-oracle が*唯一*の独立物差しである状態は禁止(§3 tripwire)。

### 分類器 parity — silent divergence の禁止(runtime 版)
研究計測器(`scripts/lib/*Classify.ts` = eventClassify / layerClassify 等)と runtime 分類器
(`src/data/ruleClassifier.ts`・`src/store/gameStore.ts` の誘発検出)は**別実装ゆえ黙って乖離する**。
研究で正した分類が runtime に入らなければ両者は別々に劣化し、§3 の鉄則「検証不能を緑に混ぜない」が
runtime 側で破れる(研究では収束、実機では誤誘発、を誰も検出しない)。規律:
- **parity テストで乖離を検出し続ける**(対応表 + 粒度差の許容差テーブル)。M0 後はコンパイラ候補として一本化を目指す。
- **乖離 = 0 を M-CONTRACT 凍結ゲート(§5-6)に接続**。粒度差で正当に一致しない対は許容差テーブルへ明示し乖離から除外。
- 計測は分類の一致だけでなく、後述の**ゴールデン再生(実行計測)**(§3 物差し束 #4)で*実際に誘発が発火するか*まで突合する。

## 4. KPI設計(スカラー正答率を捨てる)

1. **オラクル間不一致率**(構文/カード別・成熟で逓減)。**主指標。「正答率」ではない。**
2. **帰属(attribution)分布**: 各不一致を `{substrate誤り / compiler誤訳 / 物差し誤り / シナリオ曖昧}`
   へ人間が分類。`物差し誤り`の比率上昇=**物差しを直せ**の信号。`substrate誤り`が特定構文に集中=
   **ルールモデルを磨け**の信号。
3. **「ルール誤りに気づく」仕掛け**: 不一致を**共有する文法構文/ルールでクラスタリング**する。
   構文群が同じ向きに割れる→**系統的なルール誤り**。1枚だけ特異→ノイズ(オラクル誤り/曖昧)。
   集計だけでは気づけない誤謬を、構造で炙り出す。
4. **物差し校正(メタKPI)**: 人間gold部分集合の上で**各物差し自身の精度/再現率**を測る。
   LLM-oracle が尾でどれだけ外すかを数値化し、その不一致をどれだけ信じるかの重みにする。
5. **反証率**: 敵対的に N枚サンプルして**新規に見つかるモデル欠落**の発生率。逓減=corroboration。
   **ゼロや完全性は主張しない。**
6. **被覆区分(安全KPI)**: 全挙動のうち `検証可能 / 検証不能` の比率。**検証不能率 = silent divergence の上限**。

### モデル成熟KPI(M0 サイクルの収束計)
- **オントロジー被覆率**: コーパス効果行の高頻度構文のうち ESO/カタログへ写せた割合(頻度加重)。
- **モデル churn**: 反復 N→N+1 で ESO/カタログが変化した割合。**ただし下面抽出*単独*の低 churn は
  収束を意味しない** — 構造的 FN(取りこぼし)は毎反復が同一に取りこぼすため永久に churn が立たず、
  自己計測では発見できない。**収束の合図は『新しい独立物差し(盲予測/Forge差分/人間gold)を当てても
  崩れない低 churn』のみ**。
  - *precedent(2026-06-23・Slice2)*: 下面抽出単独 churn **0.05%** が「収束」を誤報したが、独立盲予測
    (203枚)を当てた瞬間 真 churn **13.55%**(2,370/17,491)が露呈した(ability-word 接頭辞
    `Landfall —`/`Revolt —` 等の構造的 FN を毎反復同一に取りこぼしていた)。
    **禁則:「低 churn ⇒ 収束」。独立物差し通過後の低 churn のみ弱い陽性。**

## 5. 収束(finishing line)と kill 基準

- **実装フェーズへの移行(凍結 = M-CONTRACT)**: 初期目標値(judgment・改訂可)。
  **下記7条件を全て満たしたとき**にのみ state モデルを §34 へ凍結し、S-* 実装に着手する。
  **Slice1+2 のみでの部分凍結は禁止**(戻せない state 設計の偽収束を防ぐ — §4 churn の precedent 参照):
  1. スライス**1〜4を一巡**(Slice3 ゾーン+プレイヤー / Slice4 タイミング+SBA を含む)。
  2. オントロジー被覆率(頭) **≥ 90%**。
  3. モデル churn **< 5%**(200枚規模の敵対サンプル)を**独立物差しを新しく当てた直後**に測って維持
     (下面抽出単独 churn では不可・§4 churn 定義)。
  4. **非LLM独立物差し**(Forge/XMage 差分 or 人間の盤面再生 gold)が代表カード集合をカバーし、不一致が有界(§3)。
  5. **ゴールデン再生ハーネス(実行計測)**が**実デッキ加重**サンプルで合格 — 分類一致ではなく
     『初期盤面+コマンド列 → 発行イベント → 誘発 → スタック解決 → SBA → 期待盤面』の盤面遷移を測る。
  6. **研究分類器 ⇄ runtime 分類器の乖離 = 0**(parity テスト緑・§3 分類器 parity)。
  7. **検証不能率**を明示公開し上限以下(silent divergence の上限を可視化)。
- **kill 基準(賭けが外れたときの逃げ道)**: 基盤(層+イベント+束縛)を入れても、**実デッキ加重**で
  反復効果の auto+guided が目標(例 70%)に届かない、または検証不能率が下がらないなら、汎用コンパイラ
  路線を止め、**デッキスコープの脚本層**(`Mydeck/` のカードに限った1枚ずつのオーバーライド=有界問題)
  へ資源を移譲する。実用上の高精度は (コンパイラの頭) ∪ (デッキ脚本) で到達可能性を担保する。

## 6. 運用と役割(トークン経済に素直)

- **メタ判断(モデル統合・物差し裁定・帰属)= Fable + ユーザー**(希少資源)。
- **機械作業(コーパス抽出・不一致リスト生成・LLM-oracle 一括予測)= Codex/スクリプト**(潤沢)。
- **1セッション = M0 の1スライス反復**。各反復の成果物は ESO/カタログの新版+不一致裁定ログ。
- 本書の手法・閾値・物差し自体も**第一級の改善対象**(初版は不完全と前提)。

## 7. オーケストレーション — 並行化と直列化の判定規則

希少資源は **Codex のスループットではなく、Fable の裁定と ESO/カタログの意味的整合性**。よって並行/直列は一点で決まる:

> **マージが安い作業は並行化せよ。マージが高い作業は直列化せよ。**

- **並行化する(独立・マージ安)**: ①独立した計測器(物差し)の構築 ②互いに素なコーパススライスの抽出
  ③敵対サンプル生成 ④`research/` への出力 ⑤**パイプライン並行**(`次の計測器を建てる Codex` ∥ `現行抽出を回す Codex`)。
  decorrelation が価値そのものの作業は積極的に別主体・別プロセスへ散らす(相関エラー遮断=§3 の orchestration 版)。
- **直列化する(結合・マージ高)**:
  - **物差しとルールの同時改変は禁止**。一度に動かす変数は1つ。物差しを動かしながらルールを直すと不一致の原因が
    「ルール改善」か「物差し変化」か区別不能になり §4-2 の帰属が死ぬ。→ **交互**:(a)物差し凍結→ルール修正→測定 /
    (b)ルール凍結→物差し改訂→再測定。**例外**: 現行物差しに触れない*新規独立物差し*の構築は並行可。
  - **ESO/カタログの統合・不一致の帰属裁定**は Fable 単独で直列(意味の衝突を避ける)。
- **マージ独占ルール**: Codex は常に*草稿/リスト*を出す。生きた2文書(ESO/カタログ)は **Fable だけが書く**。
  これが並行 Codex を安全にしている唯一の仕組み——緩めない。
- ナイーブな冗長並行(複数 Codex が同じ Fable 裁定キューに合流)はボトルネックを動かすだけで悪化する。

## 8. メタレビューと批評者プロトコル(decorrelated review)

Fable はモデルを設計し自分で監査する=盲点を共有する。これを断つため、**§3 の物差し規律を判断の層へ持ち上げる**。

### 8.1 高度(altitude)レビュー — 頻度でなくゲートで起動
- **重い高度レビューはゲート前に**:M-CONTRACT 凍結前 / スライス順の確定前 / kill-tripwire 発火時 /
  churn と executability が乖離した時(=戻せない・高コストな commit の直前)。
- **軽い定点観測は周期で**:Nループごと(初期値 = 3ループ)に短く、上の重いレビューとは別物として。
- **歯のある成果物を必須化**:振り返りは必ず **tripwire / 閾値 / ゲート条件の改訂を1つ以上**産む。数値もゲートも動かさない振り返りは禁止(=演劇)。
- **時々 decorrelated な評価者へ**:「このプログラム全体の資源配分は誤っている、と論証せよ」を *別主体*(peer級モデルの敵対プロンプト or ユーザー)に出す。自己レビューはメタレベルで盲点を共有するため。

### 8.2 批評者 = もう一つの物差し(Codex 批判レビュー)
Codex の批判的レビューは LLM-oracle と同じ物差しであり、§3 の規律を全て継承する:
1. **能力帯スコープ**: decorrelation の価値 = 独立性 × その軸での力量。Codex は**局所**(契約の自己矛盾・取りこぼしケース・
   「何をテストしていないか」・regex の明白な穴・帰属の論理破綻)に振る。**最高高度の戦略批評は peer 級モデル/ユーザー**へ。
2. **接地の強制**: 全異議に artifact(ファイル/契約条項/行)の引用 + 分類 `{correctness / consistency / scope-gap / strategic}`
