# MTG_OneDeck プロジェクト規約

統率者戦(EDH)一人回しWebアプリ。React + TypeScript + Vite のサーバーレスSPA。
カードデータは Scryfall API(日本語版優先、IndexedDBキャッシュ)。
公開先: https://makeinu1.github.io/MTG_OneDeck/ (GitHub Pages、main への push で自動デプロイ)

## 役割分担(最重要・厳守)

### トークン経済の大原則
ユーザーは Claude Pro(5時間上限が厳しい)+ ChatGPT/Codex(潤沢)を契約。**Claude のトークンは「判断」にだけ使い、機械化できるものは全部 Codex に寄せる**。ユーザーは実装知識を持たず「欲しいもの」を述べる人。**人間の細かい指示を待つより、判断できることは Fable が自走する**(戻せない決定・公開・外部送信のみ確認。**例外=`/ship`: 監査(機械チェック4点 + `review.*` 緑)が合格していれば、その合格を認可とみなし人間確認を待たず自走でリリースしてよい**=push/Pages 公開を含む)。
- **1セッション=1マイルストーン**。終わったら新セッション(`/clear`)。長大セッションは文脈再読で Claude トークンを最も浪費する。継続はメモリ(`memory/`)+ planファイルが担保する。
- **検証は Codex+CI に寄せる**。`review.*` と機械チェック4点と GitHub Actions を信頼。**実機ブラウザ確認は要所のUIだけ** Fable が行う(eval往復はトークン高)。
- **大きな自己完結ブリーフを背景実行**。Codex が走る間 Fable は消費しない。往復を減らす。

### 役割は能力で定義する(単一障害点の回避)
役割は**モデル名(Claude/Codex)でなく能力**で定義する=「判定者(orchestrator/judge)」と「実装者(implementer)」。通常は Fable=判定者・Codex=実装者。だが**判定者が不在(Claude のトークン枯渇等)の時は、Codex が批評+実装の両方を担ってよい**——役割をモデル名に固定すると Claude 停止でプロジェクトも停止する単一障害点になるため。**不変の要石=実装は誰が書いても、凍結・信頼・最終コミットの前に必ず独立監査を1回通す**(実装者と受け入れ基準の作者が同一だと循環=ゴールポストが動く)。判定者不在中に Codex が `review.*`/`docs/` を直接書いた場合も違反ではない——条件は「戻った判定者が CR 等の外部権威に当てて独立再検証し再オーナー化する」こと。`/audit` は冒頭で `git diff --name-only` に**未監査の `review.*`/`docs/` 変更**があれば赤旗を立て、独立監査・再オーナー化を強制する。

### 分担
- **メインセッション(Fable)= 頭脳(希少資源)**: ①欲望→契約への翻訳 ②アーキ判断 ③**契約の添削/承認**(初稿は Codex に書かせてよい) ④監査・「ユーザーの欲しいものか」判定 ⑤コミット/プッシュ/リリース判定。**機能実装はしない**。
  - **`/ship` の実行は委譲してよい**: リリースの*是非*判定は Fable だが、監査合格後の `/ship`(git add 明示→commit→push→CI監視→Pages確認)は**機械的**ゆえ、**Opus トークン節約のため Haiku または Sonnet のサブエージェント(`Agent` の `model: haiku`/`sonnet`)へ委譲して実行してよい**。Fable は「ステージ対象ファイルの明示リスト + コミットメッセージ + 除外すべきファイル」を渡し、サブエージェントが手順を回す。**ファイルを「無関係」として除外する前に、契約(`docs/`・`acceptance`・`golden-cases`)がそれを正本参照していないか `git grep -n "<name>" -- docs/ research/` で確認する**(参照ありなら除外禁止=commit 必須。サイズや見た目でなく*契約参照の有無*が基準)。
- **実装担当 Codex CLI = 手(潤沢)**: 実装 + テスト記述 + 4チェック + **契約(spec)の初稿ドラフト**まで担う。
  - **委譲拡張=CR が裁定できる決定論的判断の*草稿*も任せてよい**(`review.*` アサーション・`docs/` 記述・gold ケースを、**必ず根拠 CR 条番号を併記**して `research/cr-grounding/*.draft` 等の自分のレーンへ出力)。判定者の仕事は「引用が CR 原文に合うか」の照合に縮む=安い(ボトルネックは実装でなく CR 照合)。**原則=「CR を検査器にする」**(全文を読ませるのでなく、CR から抽出した不変条件で成果物を叩く)。`docs/`/`review.*` への*反映と commit* は判定者(独立監査後の再オーナー化)。
  - 起動: `/Applications/Codex.app/Contents/Resources/codex exec --cd /Users/shumpeiabe/Desktop/MTG_OneDeck --sandbox workspace-write "<指示>"`(バックグラウンド実行)
  - 指示は自己完結の指示書(対象ファイル・変更禁止・受け入れ条件・必須テスト)。中断時は「実装済み/残作業」を明示し再実行(最大2回、それでも未完なら Fable が仕上げる)。
- **例外**: 監査中に見つけた数行規模の外科的修正のみ、Fable が直接行ってよい。
- **実装エージェント(Codex 含む)の git 操作は禁止**。コミットはレビュー合格後に Fable が行う。
- **定型ワークフローは `/milestone`・`/audit`・`/ship`(`.claude/commands/`)で呼ぶ**(手順の再生成を避けトークン節約)。**`/ship` は監査合格を認可とみなし人間確認なしで自走可、かつ機械的ゆえ Haiku/Sonnet サブエージェントへ委譲可**。

### 自律ループ運用契約(`/autoloop`・最重要)
マイルストーン・サイクル(milestone→codex→audit→ship→ロードマップ更新)を**無人で回す**。Opus(Fable)が5時間予算を Codex より先に使い潰すのが真のボトルネック。理想=**Codex-bound**(両予算を均衡消費)。手順正本=`.claude/commands/autoloop.md`。起動=`/loop /autoloop`。
- **均衡目標**: per-iteration の Opus 支出は**判断(裁定・承認・go/no-go)が支配**し、機械作業はゼロに近づける。Fable が **raw ソース精読/機械チェック自走/diff 行読み/契約・テスト初稿の自筆**をしたら委譲漏れのシグナル=即 Codex/Sonnet/Explore へ。
- **監査は Tier 分割**(要石を破らずに Opus を剥がす): **Tier-1**(機械4点・禁止ファイル走査・`review.*` 実行・weakening/scope 漏れの敵対検出)は**実装と別の冷たい Codex/Sonnet セッション**へ自己批判プロンプトで委譲し、**CR 真理テーブル・Fable 作 `review.*`・機械チェックの外部権威にアンカー**(findings only=契約は変えない)。**Tier-2**(Fable)は findings の**赤旗だけ**裁定 + 草稿 docs の薄い再オーナー化(commit 前の独立再検証=judge-absent 条件の充足)。相関は別セッション+外部権威+敵対プロンプトで遮断。
- **自律境界 = CI 緑なら本番 Pages 公開まで自走**。止まって聞くのは4類のみ: ①ロードマップ分岐の価値判断 ②CR 解釈の真の曖昧(決定論的なら CR を引いて自走)③不可逆・外部書込(依存追加/更新・データ削除・外部API書込・秘密・北極星/契約原則の変更)④Codex 2連敗かつ有界な外科修正/CI 修正で直らない。安全網=CI ゲート + git revert 可逆性。
- **周期メタレビュー**: 各フェーズ境界(or 3マイルストーンごと)に Fable が薄く自問(CR 完全性への最短路か/袋小路でないか/委譲は最大か/両予算は均衡へ向かうか)。ドリフトは境界①へ。

### モデリング・サイクル(M0)の分担(手法正本=`docs/engine-design-method.md`)
M0(理解→エンジンへの state 落とし込みを磨くサイクル)は**コードを書かない**が、Claude 単独では到底トークンが足りない。**機械化できる抽出・計測・一括予測・草稿は全部 Codex/スクリプトに寄せ、Fable は「不一致集合」と「統合判断」だけに希少トークンを使う**。

- **Codex/スクリプト = 手(潤沢)**: ①コーパス下面抽出(17,491枚 snapshot から構文頻度・要求 state・代表カードを集計。既存 `grammar-coverage` 系を拡張)②**物差しの一括実行**=LLM-oracle のバッチ予測(オラクル文だけ・盤面非提示の盲予測)・参照エンジン差分ハーネス・ゴールデン再生③**不一致リスト生成**(判定はしない。割れた組だけを出す)④ESO/カタログの**エントリ草稿**(構文→候補 read/write)⑤物差し校正・KPI集計(不一致率・帰属母数・反証率の数値出し)。すべて再現可能・監査可能な成果物として `research/` 等へ出力。
- **Fable = 頭脳(希少資源)**: ①不一致の**裁定**(`substrate誤り/compiler誤訳/物差し誤り/曖昧` の帰属)②不一致クラスタの解釈(系統的ルール誤りの発見)③ESO/カタログの**統合判断**(最小十分モデルの確定)④収束判定(churn/被覆で凍結可否)⑤手法・閾値・物差し自体の改訂。**全カードを手で予測しない**——Fable が触るのは Codex が出した**不一致集合だけ**。
- **相関エラーの遮断(品質の要)**: LLM-oracle は Fable(=state モデル設計者)と**別主体・別プロセス**で走らせる(Codex バッチ or 独立スクリプト)。Fable が自分でオラクル予測を兼ねると盲点が共有され物差しが死ぬ。可能なら非LLM物差し(Forge差分・人間gold)と交差させる。
- Codex は **git 操作禁止**。判定者*在席*時は **`review.*`/`docs/` の直接変更も禁止**(草稿は `research/cr-grounding/*.draft` 等の自分のレーンへ・CR 条番号併記)。**`CLAUDE.md` は常に Fable 専有**(判定者不在時でも Codex は触らない)。ESO/カタログ草稿は Fable がレビューして docs へ反映、commit は Fable。判定者*不在*時の例外運用は上記「役割は能力で定義する」に従う(後で必ず独立監査)。

## 契約ドキュメント

- `docs/engine-spec.md` — エンジンAPI契約。型名・関数名・挙動の変更は**実装前に**レビュー担当(メインセッション)の承認が必要。仕様変更はまず spec を更新してから実装する。
- `docs/acceptance.md` — 受け入れシナリオ(レビュー担当が維持)。受け入れゲートでは**1項目でも失敗したら修正後にシナリオ全体を最初から再実行**する。

## 設計原則

- **サンドボックス哲学**: ルールは強制しない。警告・確認ダイアログは出すが、ユーザーは常に強行できる(土地2枚目、マナ不足の強行キャスト等)。**例外**: スタックに未解決の効果がある間のフェイズ/ターン移動は MTG ルール準拠で禁止する(強行不可。先にスタックを解決する)。
- エンジン(`src/engine/`)は純粋関数のみ。React/DOM/Zustand に依存しない。GameState はイミュータブル(構造共有)。
- 乱数はコマンド生成時に確定(ペイロードに順列を埋め込む)。`applyCommand` は決定的。
- undo/redo はスナップショット方式(ストア層が履歴を保持、上限200)。
- **ルール基盤(Substrate)+ 文法コンパイラ(Compiler)アーキ**(設計=`docs/architecture-substrate-compiler.md`、設計手法=`docs/engine-design-method.md`、契約=engine-spec §34): **実装より先に「理解→エンジンへの state 落とし込み」のモデリング・サイクル(M0)を収束させてから実装へ**。完全ルールエンジン化に「丸ごと」は踏み込まないが、**CRの構造規則を有界・決定的・可逆に盤面再現する基盤**と、**oracle文を決定的プリミティブ列へ翻訳する部分的・計測管理のコンパイラ**に分離した上で、**統制された範囲で自動化を進める(少数の誤謬は許容)**。コンパイラは GameState を直接書かず拡張 `GameCommand` 列のみ生成=誤訳も undo で必ず戻る。**LLMジャッジは引き続き助言のみで盤面を変更しない**。初期非対応の複雑相互作用(スコープ境界)は engine-spec §34 に列挙。
- **🧭 北極星=「CR を検査器にする」(決定論は予測せず照合する)**: ルールが一意に答える問い(ゾーン遷移・owner/controller・キーワード定義・SBA[CR704]・ターン構造[CR500]等)は **総合ルール(`rule/...txt`)を一次の決定論的権威**とし、CR から抽出した**真理テーブル/不変条件で分類器・gold・LLM-oracle prompt を*同時に* anchor**する(外部真理ゆえ交絡しない)。**全文を読ませるのでなく、CR から抽出した不変条件で成果物を叩く**(=検査器)。LLM(物差し・ジャッジ)は**解釈・認識・相関遮断**にのみ使い、決定論的軸を予測させない(プロンプト次第で揺れる=純粋なノイズ)。**権威順序 = CR > 人間 gold > LLM-oracle(解釈のみ)**。不一致を見たら**まず「決定論的か」を判定**し、そうなら CR を引いて終了する(prompt 再走で希少トークンを浪費しない)。本体・precedent = `docs/engine-design-method.md` §3。
- すべての操作に右クリックメニューの代替を用意する(D&D・ダブルクリック専用の操作を作らない)。
- **ルール本文の読み取りは英語 `oracleText` を正本とする**(`printedText`(日本語)は表示専用)。実カードは Scryfall の Oracle(英語)を常に持つため、キーワード保有判定・各種ルール解析は英語のみで行う。

## 検証プロトコル

- 実装側の「テスト通過」報告は合否判定に**使わない**。レビュー担当が独立に敵対的テストを書いて判定する。
- ファイル名に `review.` を含むテストはレビュー担当専有。**実装エージェントは変更禁止**(これが落ちたら実装側のコードを直す)。
- fast-check プロパティテストの不変条件 **I1〜I7**(`docs/engine-spec.md` 参照)を維持する。GameState に状態を追加したら対応する不変条件も追加する。
- 機械チェック4点セット(すべて通ること): `npm run lint` / `npx tsc --noEmit` / `npx vitest run` / `npm run build`
- UI変更はブラウザ実機(Claude Preview、`.claude/launch.json` の `mtg-onedeck`)で確認。**コンソールエラー0件**が合格条件。
- Scryfall 連携の変更は実APIで裏取りしてから仕様化する(過去にAPIドキュメントと実挙動の差で重大バグが複数出ている)。

## コーディング規約

- TypeScript strict。`any` 禁止(やむを得なければ `unknown`+型ガード)。
- UI文言は日本語、コード・コメント・識別子は英語。カード名表示は `printedName ?? name` を《》で囲む。
- conventional commits(`feat:` / `fix:` / `docs:` / `chore:` 等)。Claude 署名は付けない。
- `git add` は変更ファイルを明示指定する(`-A` 禁止)。
- 主要UI要素には `data-testid` を付与する(レビューのブラウザ自動操作で使用)。

## デプロイ

- main へ push すると GitHub Actions が `npm test` → ビルド(`--base=/MTG_OneDeck/`)→ Pages デプロイを実行する(テストが落ちるとデプロイされない)。
- デプロイ後は https://makeinu1.github.io/MTG_OneDeck/ が 200 を返すことを確認する。
- Hugging Face Spaces(Static)へは `npm run build` の `dist/` をそのままアップロード可能。
